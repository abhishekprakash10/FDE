Tech Stack

ServiceDesk Pro is built as a single-page React application with a Node.js/Express backend and a PostgreSQL database. The stack was selected to balance developer velocity, type safety, and operational simplicity for an internal tooling context.

Layer Technology Version Purpose
Frontend Framework React 18.2 Component-based UI rendering with hooks for state management
Language TypeScript 5.3 Static typing across frontend and backend for reliability
Styling Tailwind CSS 3.4 Utility-first CSS for rapid UI development
Backend Framework Node.js + Express 20 LTS / 4.18 RESTful API server handling business logic and data access
Database PostgreSQL 16 Relational data store for tickets and activity logs
ORM Prisma 5.8 Type-safe database queries and schema migrations
Authentication JWT (jsonwebtoken) 9.0 Stateless session management with role claims
Build Tool Vite 5.1 Fast HMR development server and production bundler
HTTP Client Axios 1.6 Promise-based API requests from the React frontend
Testing Vitest + Testing Library 1.3 Unit and integration testing for components and API routes
Deployment Docker + Compose 25 Containerized deployment for consistent environments

API Reference

All endpoints are prefixed with /api/v1. Authenticated endpoints require a Bearer JWT token in the Authorization header. Responses follow the format { success: boolean, data: T, message: string }.

Method Path Auth Required Description
POST /auth/login No Authenticate a user by name and role. Returns a signed JWT containing userId, name, and role claims. Token expires in 8 hours.
GET /tickets Yes Retrieve all tickets. Supports query params: q (keyword), category, status, priority. Returns paginated list ordered by created_at DESC.
GET /tickets/:id Yes Retrieve a single ticket by its numeric ID, including all associated activity log entries ordered chronologically.
POST /tickets Yes Create a new support ticket. Required body fields: employee_name, department, category, description. Optional: priority (defaults to "Medium"). Automatically sets status to "Open".
PUT /tickets/:id Yes — Admin only Update an existing ticket's status, priority, and/or resolution_notes. Requires changed_by in the body. Automatically creates an activity log entry recording the before/after state.
DELETE /tickets/:id Yes — Admin only Permanently delete a ticket and all its activity log entries (cascading delete). This action is irreversible. Returns 204 No Content on success.
GET /tickets/:id/activities Yes Retrieve the full activity log for a specific ticket, ordered by changed_at ASC. Each entry includes the actor name, previous status, new status, and ISO timestamp.

Database Schema

The application uses two primary tables. The tickets table holds the canonical ticket state; the ticket_activities table provides an append-only log of all mutations, enabling full audit capability.

Table: tickets
Column Type Constraints Description
id SERIAL PRIMARY KEY Auto-incrementing unique ticket identifier
employee_name VARCHAR(100) NOT NULL Full name of the employee who raised the ticket
department VARCHAR(100) NOT NULL Department of the reporting employee
category VARCHAR(100) NOT NULL Issue type: VPN Issue, Password Reset, Software Installation, Laptop Issue, Email Access, Other
priority VARCHAR(20) NOT NULL DEFAULT 'Medium' Ticket urgency: Low, Medium, High, Critical
status VARCHAR(20) NOT NULL DEFAULT 'Open' Current lifecycle state: Open, In Progress, Resolved, Rejected
description TEXT NOT NULL Full description of the issue as entered by the employee
resolution_notes TEXT NULLABLE Admin-entered notes explaining how the issue was resolved or why it was rejected
created_at TIMESTAMPTZ DEFAULT NOW() UTC timestamp of ticket creation
updated_at TIMESTAMPTZ DEFAULT NOW() UTC timestamp of last modification; auto-updated via trigger
Table: ticket_activities
Column Type Constraints Description
id SERIAL PRIMARY KEY Auto-incrementing log entry identifier
ticket_id INTEGER FK → tickets.id ON DELETE CASCADE Reference to the parent ticket; cascades on deletion
changed_by VARCHAR(100) NOT NULL Name of the admin or actor who performed the change
prev_status VARCHAR(20) NULLABLE Status before this change; NULL for the initial creation entry
new_status VARCHAR(20) NOT NULL Status after this change
prev_priority VARCHAR(20) NULLABLE Priority before this change; NULL if priority was not modified
new_priority VARCHAR(20) NULLABLE Priority after this change; NULL if priority was not modified
notes TEXT NULLABLE Optional free-text note captured at the time of the change
changed_at TIMESTAMPTZ DEFAULT NOW() UTC timestamp of when the change was recorded

Roles & Permissions

ServiceDesk Pro enforces role-based access control at both the API layer and UI layer. There are two roles: Employee (self-service, read-oriented) and IT Admin (full ticket lifecycle management). Role is embedded in the JWT and validated on every protected endpoint.

Action Endpoint Employee IT Admin Notes
Login / Get Token POST /auth/login ✓ ✓ Both roles authenticate via the same endpoint with role selection
View all tickets GET /tickets ✓ ✓ All authenticated users can browse the full ticket list
View ticket detail GET /tickets/:id ✓ ✓ All authenticated users can view a specific ticket and its metadata
Raise new ticket POST /tickets ✓ ✓ Any authenticated user can create a support request
Update ticket status / priority PUT /tickets/:id ✗ ✓ Employees receive 403 Forbidden; change triggers an activity log entry
Delete ticket DELETE /tickets/:id ✗ ✓ Permanently removes ticket and all activity logs; irreversible
View activity log GET /tickets/:id/activities ✓ ✓ Full audit trail is visible to all authenticated users for transparency
Search & filter tickets GET /tickets?q=&category=&status= ✓ ✓ Query parameter filtering is available to all authenticated users
View Edit / Delete UI controls Frontend only ✗ ✓ Edit and Delete buttons are conditionally rendered based on the JWT role claim
See employee info banner Frontend only ✓ ✗ The read-only access reminder banner is shown only to Employee role users
